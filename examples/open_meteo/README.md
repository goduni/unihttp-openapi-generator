# Open-Meteo — async + msgspec client + concurrent live runner

A typed [unihttp](https://github.com/goduni/unihttp) client for the
[Open-Meteo](https://open-meteo.com) weather API, generated by
`unihttp-openapi-generator`, plus an **async** runner that fetches several
cities **concurrently** and asserts the results.

| | |
|---|---|
| **Serializer** | msgspec |
| **Client** | async, httpx |
| **Layout** | flat (`await client.get_forecast(...)`) |
| **File layout** | per-object (one file per model / method) |
| **Auth** | none |

Generation settings live in
[`unihttp-openapi-generator.toml`](unihttp-openapi-generator.toml) — the
generator auto-discovers it, so `regen.sh` carries no flags.

What this example shows that [frankfurter](../frankfurter/) doesn't: an **async**
client, **concurrent** requests via `asyncio.gather`, the **msgspec** serializer,
and a response with **nested structs + parallel arrays**.

## Run it

```bash
cd open_meteo_runners
uv run python weather_demo.py
```

Sample output:

```

City        Temp °C  Wind km/h  Code
Berlin         22.1       11.3     3
London         23.3       10.8     3
Tokyo          21.1        1.9     1
New York       30.9       12.2     0
Sydney         11.0        7.2     2
...
OK — forecast endpoint exercised (current + hourly), 5 cities fetched concurrently, all assertions passed.
```

Exit codes: `0` all assertions passed · `1` an assertion failed (a real bug) ·
`2` the API was unreachable (skipped, not a bug).

```bash

```

## Endpoint coverage

The client has a single endpoint, exercised in both of its modes:

| Call | Endpoint | What the runner checks |
|---|---|---|
| `get_forecast(current=...)` | `GET /v1/forecast` | 5 cities concurrently; temp in [-60, 60]°C, wind ≥ 0, integer weather code, coordinate echoed back |
| `get_forecast(hourly=...)` | `GET /v1/forecast` | `time` and `temperature_2m` arrays line up element-for-element |

Live weather is not deterministic, so the assertions are structural and
sane-range rather than exact (unlike the frankfurter example's fixed historical
rates).

## Note on the spec

Open-Meteo does not publish an OpenAPI document, so `openapi.yaml` is
**hand-authored** to match the live `/v1/forecast` endpoint. Open-Meteo's
response shape depends on the variables requested via `current=` / `hourly=`;
the spec models exactly the fields the runner asks for (`temperature_2m`,
`wind_speed_10m`, `weather_code`), and `current` / `hourly` are optional since
each is only present when requested. `msgspec` ignores any extra fields the API
returns, so modelling a subset is safe.
