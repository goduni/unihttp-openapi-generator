# Frankfurter — generated client + live runner

A typed [unihttp](https://github.com/goduni/unihttp) client for the
[Frankfurter](https://frankfurter.dev) foreign-exchange API, generated by
`unihttp-openapi-generator`, plus a runner that calls the **live** API and
asserts the results.

| | |
|---|---|
| **Serializer** | adaptix |
| **Client** | sync, httpx |
| **Layout** | flat (`client.get_*` directly on the client) |
| **File layout** | per-object (one file per model / method) |
| **Auth** | none |

Generation settings live in
[`unihttp-openapi-generator.toml`](unihttp-openapi-generator.toml) — the
generator auto-discovers it, so `regen.sh` carries no flags.

## Layout

```
frankfurter/
├── openapi.yaml                      # vendored OpenAPI 3.1 spec (see note below)
├── unihttp-openapi-generator.toml    # generation settings (auto-discovered)
├── regen.sh                          # thin wrapper: generate + ruff/mypy gate
├── frankfurter_library/              # the GENERATED, installable client package
│   └── frankfurter_client/
│       ├── models/                   # one file per model (per-object layout)
│       │   ├── exchange_rates.py
│       │   ├── time_series_rates.py
│       │   └── currency_map.py
│       ├── methods/default/          # one file per operation
│       │   ├── get_latest_rates.py
│       │   └── ...
│       ├── _forward_refs.py          # resolves cross-module model references
│       ├── client.py
│       └── _serialization.py
└── frankfurter_runners/              # hand-written demo + assertions
    └── rates_demo.py
```

## Regenerating

```bash
./regen.sh        # runs `unihttp-openapi-generator generate` + ruff/mypy
```

The command takes no flags: every setting (spec, package name, serializer,
backend, layout, file layout) is read from `unihttp-openapi-generator.toml`.

## Run it

```bash
cd frankfurter_runners
uv run python rates_demo.py
```

`uv` resolves the generated library next door via an editable path source (see
`frankfurter_runners/pyproject.toml`), so there is nothing to install by hand.

Sample output:

```
Frankfurter client -> https://api.frankfurter.dev

=== Historical rates for 2020-01-02 (base USD) ===
base=USD  date=2020-01-02  rates={'EUR': 0.89342, 'GBP': 0.75787}
...
OK — 5/5 endpoints exercised, all assertions passed.
```

Exit codes: `0` all assertions passed · `1` an assertion failed (a real bug) ·
`2` the API was unreachable (skipped, not a bug).

## Endpoint coverage

The runner exercises **every** endpoint of the Frankfurter API:

| Endpoint | Method | What the runner checks |
|---|---|---|
| `get_latest_rates` | `GET /v1/latest` | default base is EUR, non-empty `float` rates |
| `get_rates_for_date` | `GET /v1/{date}` | **exact** historical rates (deterministic), `symbols` CSV filter, amount scaling |
| `get_time_series` | `GET /v1/{start}..{end}` | nested `{date: {code: rate}}` map; `2020-01-02` GBP is exact |
| `get_time_series_to_now` | `GET /v1/{start}..` | open-ended series, first working day present |
| `get_currencies` | `GET /v1/currencies` | known codes, `USD == "United States Dollar"` |

The historical-rate assertions are exact (`2020-01-02` USD→GBP is always
`0.75787`) — past reference rates never change, which makes this the strongest
possible proof the client really round-trips the API.

Notable shapes exercised by generation:

- `rates: dict[str, float]` and the time-series `dict[str, dict[str, float]]`
  (`additionalProperties` maps, nested);
- a date-range path with **two path params in one segment**
  (`/v1/{start_date}..{end_date}`), plus the open-ended `/v1/{start_date}..`;
- `symbols` as a `list[str]` serialized to a single CSV query param.

## Note on the spec

Frankfurter does not publish an OpenAPI document, so `openapi.yaml` here is
**hand-authored** to match the live API. It is kept deliberately small — just the
three no-auth `GET` endpoints the runner exercises. Everything else (models,
request classes, the client, serialization wiring) is generated from it by
`regen.sh`.
