# JSONPlaceholder — grouped client over a multi-resource REST API

A typed [unihttp](https://github.com/goduni/unihttp) client for
[JSONPlaceholder](https://jsonplaceholder.typicode.com), generated by
`unihttp-openapi-generator`, plus a runner that calls the **live** API and
asserts the results.

| | |
|---|---|
| **Serializer** | adaptix |
| **Client** | sync, **requests** (not httpx) |
| **Layout** | **grouped** (`client.posts.*`, `client.users.*`, …) |
| **File layout** | per-object |
| **Auth** | none |

Generation settings live in
[`unihttp-openapi-generator.toml`](unihttp-openapi-generator.toml).

What this example adds over the others: a **grouped** client across four tags, a
non-httpx sync backend (**requests**), **nested models** (`User → Address → Geo`
/ `Company`), and **camelCase query-parameter aliasing** (`?userId=`, `?postId=`).

## Run it

```bash
cd jsonplaceholder_runners
uv run python reads_demo.py
```

Sample output:

```
=== posts ===
post 1: user_id=1 title='sunt aut facere repellat provident occae'...
100 posts total, 10 by user 1
...
=== users ===
user 1: Leanne Graham <Sincere@april.biz> in Gwenborough @ Romaguera-Crona
...
created -> id=101 user_id=7 title='hello'
...
OK — 11 endpoints across 4 tags exercised (incl. create/update/delete), all assertions passed.
```

## Endpoint coverage

| Tag | Endpoints | What the runner checks |
|---|---|---|
| posts | `list_posts`, `get_post`, `create_post`, `update_post`, `delete_post`, `get_post_comments` | post 1 fixtures exact; `?userId=1` filter returns 10; create echoes `id=101` with `userId` intact; 5 comments |
| comments | `list_comments` | `?postId=1` returns 5 comments |
| users | `list_users`, `get_user` | nested `address.city`, `address.geo.lat`, `company.name`; 10 users |
| todos | `list_todos`, `get_todo` | todo 1 fixtures exact; `?userId=1` filter |

The write endpoints take the body as keyword arguments
(`create_post(user_id=7, title=…, body=…)`), because an object request body is
spread into individual `Body` fields — unihttp keys each `Body`-marked field into
the JSON body, so the client POSTs `{"userId": 7, …}` directly.

## Why adaptix

JSONPlaceholder's query filters and body use camelCase (`userId`, `postId`).
adaptix aliases request parameters and body fields via `name_mapping`; the
pydantic/msgspec request dumpers currently key query/path/header/body values by
the Python field name and don't alias them, so `?userId=` would be sent as
`?user_id=` and silently ignored. For a camelCase API, generate with adaptix.

JSONPlaceholder does not publish an OpenAPI document, so `openapi.yaml` is
hand-authored to match the live API.
